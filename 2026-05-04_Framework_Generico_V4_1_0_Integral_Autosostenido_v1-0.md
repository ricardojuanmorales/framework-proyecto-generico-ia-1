# 0. Declaración de autosuficiencia

Esta versión del **Framework Genérico V4.1.0** está diseñada para ser **autosostenida**.

Esto significa que una persona o equipo puede:

```text
leer este documento;
identificar sus necesidades;
determinar nivel N1-N4;
activar perfil o Caleidoscopio;
definir alcance;
crear ruta de trabajo;
aplicar Gates y CHECKS;
documentar decisiones;
evaluar avances;
cerrar sesión;
archivar resultados;
preparar continuidad.
```

No requiere leer V4.0.0 para operar.  
No requiere documentos externos para iniciar.  
Puede conectarse a repositorios, wikis, Obsidian, GitHub, Drive, Notion u otros sistemas, pero no depende de ellos.

# 1. Resumen ejecutivo

El **Framework Genérico V4.1.0** es una arquitectura de trabajo humano-IA para diseñar, investigar, crear, programar, documentar, evaluar, publicar, archivar y evolucionar proyectos complejos.

Su enfoque combina:

```text
juicio humano;
inteligencia artificial;
ética;
accesibilidad;
documentación activa;
evaluación;
gobernanza;
archivo histórico;
continuidad;
transferencia.
```

Puede usarse en proyectos educativos, investigativos, artísticos, tecnológicos, comunitarios, institucionales, administrativos, curriculares, culturales o de innovación.

El framework no es una plantilla rígida. Es una brújula con cajas de herramientas.

# 2. ¿Para qué sirve?

Sirve para organizar proyectos que necesitan una o más de estas capacidades:

```text
entender una necesidad compleja;
diseñar un proyecto;
integrar IA responsablemente;
crear materiales, productos o sistemas;
investigar con evidencia;
programar o prototipar;
desarrollar experiencias artísticas o multimodales;
documentar decisiones;
trabajar con equipos humanos e IA;
evaluar calidad;
garantizar accesibilidad;
gestionar riesgos;
crear continuidad;
archivar resultados;
preparar transferencia o adopción.
```

# 3. Principios rectores

## 3.1 Juicio humano

La IA asiste, pero no sustituye la responsabilidad humana.

```text
La IA propone.
La persona evalúa.
El equipo decide.
La documentación recuerda.
```

## 3.2 Ética y cuidado

Todo proyecto debe considerar impactos sobre personas, comunidades, datos, cultura, ambiente, derechos y accesibilidad.

## 3.3 Documentación activa

La documentación no ocurre al final. Acompaña el trabajo.

```text
Si se decide, se registra.
Si se cambia, se documenta.
Si se cierra, se archiva.
Si se continúa, se deja primer.
```

## 3.4 Escala proporcional

No todo proyecto necesita todo el framework.

```text
N1 para tareas pequeñas.
N2 para proyectos operativos.
N3 para proyectos complejos.
N4 para institucionalización.
```

## 3.5 Cierre y continuidad

Una sesión no termina cuando se acaba la conversación. Termina cuando queda claro:

```text
qué se hizo;
qué cambió;
qué falta;
dónde está;
quién puede retomarlo;
cuál es el próximo paso.
```

El Framework Genérico V4.1.0 contiene cinco componentes principales:

# 5. Protocolo de identificación de necesidades del usuario

Antes de iniciar un proyecto, aplicar este diagnóstico.

## 5.1 Preguntas iniciales

Responder brevemente:

```text
1. ¿Qué problema, necesidad o deseo quieres atender?
2. ¿Quiénes son las personas o comunidades afectadas?
3. ¿Qué producto, resultado o cambio esperas lograr?
4. ¿El proyecto requiere investigación, creación, programación, enseñanza, gestión o integración?
5. ¿Usarás IA? ¿Para qué?
6. ¿Hay datos sensibles, comunidades vulnerables, menores, imágenes, voces, historias personales o patrimonio cultural?
7. ¿El resultado será privado, interno, académico, público o institucional?
8. ¿Cuánto tiempo durará el proyecto?
9. ¿Trabajará una persona, un equipo pequeño, una comunidad o una institución?
10. ¿Qué evidencia demostrará que hubo avance?
11. ¿Qué riesgos te preocupan?
12. ¿Quién debe poder retomar el trabajo después?
```

## 5.2 Clasificación rápida de necesidad

| Respuesta dominante | Tipo de necesidad |
|---|---|
| Quiero construir, automatizar, programar, prototipar o manejar repositorio | Técnica / Programador Humanista |
| Quiero investigar, analizar fuentes, producir conocimiento o evidencia | Investigativa / Investigador Transdisciplinario |
| Quiero crear obra, narrativa, avatar, experiencia, galería o material multimodal | Creativa / Artista Transdisciplinario |
| Quiero integrar investigación, programación, creación, educación o comunidad | Integrada / Caleidoscopio |
| Quiero institucionalizar, escalar, capacitar o gobernar adopción | Institucional / N4 |

# 6. Determinar nivel N1-N4

## 6.1 Tabla rápida

| Nivel | Nombre | Usar cuando | Documentación mínima |
|---|---|---|---|
| N1 | Mínimo controlado | tarea breve, bajo riesgo, una persona | propósito, producto, cierre breve |
| N2 | Operativo | proyecto con varias fases o equipo pequeño | README, checklist, decisiones básicas, cierre |
| N3 | Avanzado | integración, IA sustantiva, riesgos, wiki, archivo | Gates, CHECKS, changelog, decisiones, primer |
| N4 | Institucional | adopción formal, política, auditoría, capacitación | gobernanza formal, RACI, auditoría, mantenimiento |

## 6.2 Reglas para subir de nivel

Subir de N1 a N2 si:

```text
hay más de una sesión;
hay equipo;
hay documento reusable;
hay uso frecuente de IA;
hay necesidad de seguimiento.
```

Subir de N2 a N3 si:

```text
hay perfiles transversales;
hay Caleidoscopio;
hay publicación;
hay datos o comunidad sensible;
hay riesgos éticos, culturales o socioambientales;
hay archivo o wiki;
hay necesidad de trazabilidad.
```

Subir de N3 a N4 si:

```text
hay adopción institucional;
hay capacitación formal;
hay auditoría;
hay mantenimiento asignado;
hay política o programa;
hay impacto sobre muchas personas.
```

## 6.3 Reglas para bajar de nivel

Bajar de nivel si:

```text
la tarea es exploratoria;
no hay publicación;
no hay equipo;
no hay datos sensibles;
no hay integración compleja;
no habrá mantenimiento.
```

# 7. Perfiles Transversales

Los perfiles son lentes operativos. Ayudan a saber qué tipo de trabajo domina.

---

## 7.1 Programador Humanista

### Función

Construir sistemas, scripts, prototipos, flujos técnicos, repositorios, automatizaciones y productos digitales con responsabilidad humana, documentación y evaluación.

### Activar cuando el proyecto requiere:

```text
código;
arquitectura técnica;
repositorio;
prototipo;
automatización;
datos;
pruebas;
seguridad;
mantenimiento;
documentación técnica.
```

### Evidencias típicas

```text
README técnico;
código;
diagrama;
script;
tests;
checklist;
release;
documentación de instalación;
registro de decisiones técnicas.
```

### Riesgos típicos

```text
automatización prematura;
seguridad débil;
dependencias no documentadas;
datos sensibles expuestos;
código sin pruebas;
IA usada sin revisión.
```

---

## 7.2 Investigador Transdisciplinario

### Función

Investigar problemas complejos mediante fuentes, métodos, evidencia, análisis crítico, integración de saberes y transferencia.

### Activar cuando el proyecto requiere:

```text
pregunta de investigación;
revisión de literatura;
fuentes;
bibliografía;
evidencia;
marco conceptual;
análisis;
publicación;
ciencia abierta;
REA;
transferencia educativa.
```

### Evidencias típicas

```text
pregunta guía;
matriz de fuentes;
evidence ledger;
síntesis;
bibliografía;
marco conceptual;
metodología;
limitaciones;
producto de divulgación.
```

### Riesgos típicos

```text
fuentes débiles;
sesgos;
extracción epistémica;
IA resumiendo sin verificación;
confusión entre dato e interpretación;
falta de citación.
```

---

## 7.3 Artista Transdisciplinario

### Función

Crear, curar, documentar y publicar obras, experiencias, narrativas, avatares, colecciones, memoria, materialidades y productos multimodales.

### Activar cuando el proyecto requiere:

```text
creación artística;
narrativa;
avatar;
experiencia inmersiva;
galería;
curaduría;
memoria;
territorio;
símbolos;
materialidades;
derechos culturales;
publicación multimodal.
```

### Evidencias típicas

```text
statement;
boceto;
prototipo;
obra;
ficha curatorial;
licencia;
registro de derechos;
archivo de obra;
documentación de proceso.
```

### Riesgos típicos

```text
apropiación cultural;
derechos de autor;
representación inadecuada;
uso de IA generativa sin disclosure;
archivo incompleto;
accesibilidad estética débil.
```

# 8. Caleidoscopio

El Caleidoscopio no es un cuarto perfil tradicional. Es la lógica integradora que se activa cuando un proyecto requiere combinar perfiles para producir algo emergente.

## 8.1 Pregunta clave

```text
¿Qué aparece cuando programación, investigación y creación se transforman mutuamente bajo evidencia, ética, accesibilidad, documentación y transferencia?
```

## 8.2 Cuándo activar Caleidoscopio

Activar cuando:

```text
un solo perfil no basta;
hay integración entre investigación, creación y técnica;
hay producto multimodal;
hay comunidad o institución;
hay necesidad pedagógica;
hay archivo vivo;
hay transferencia;
hay complejidad ética, cultural o socioambiental.
```

## 8.3 Modos C1-C7

| Modo | Combinación | Uso típico |
|---|---|---|
| C1 | Programador + Investigador | apps de investigación, dashboards, bases de datos, pipelines |
| C2 | Investigador + Artista | investigación-creación, memoria, publicaciones multimodales |
| C3 | Artista + Programador | avatares, instalaciones, interfaces, galerías digitales |
| C4 | Tres perfiles | AVAT, REA multimodal, ecosistemas agénticos |
| C5 | Pedagógico | cursos, módulos, OVAT, trayectorias formativas |
| C6 | Comunitario | archivo vivo, mapa territorial, memoria colectiva |
| C7 | Institucional | laboratorios, programas, políticas, manuales, wiki institucional |

# 9. Flujo general de implementación

## 9.1 Fase 1 — Activación

```text
1. Describir necesidad.
2. Identificar contexto.
3. Determinar nivel N1-N4.
4. Activar perfil o Caleidoscopio.
5. Definir producto esperado.
6. Identificar riesgos iniciales.
7. Escoger cartapacio o espacio de trabajo.
```

## 9.2 Fase 2 — Diseño

```text
1. Crear brief del proyecto.
2. Definir alcance.
3. Seleccionar metodología.
4. Definir evidencia mínima.
5. Definir roles humanos e IA.
6. Crear ruta de trabajo.
```

## 9.3 Fase 3 — Producción

```text
1. Crear documento, prototipo, obra, matriz, código o producto.
2. Documentar decisiones.
3. Registrar uso de IA.
4. Revisar accesibilidad.
5. Revisar ética y derechos.
6. Aplicar CHECKS.
```

## 9.4 Fase 4 — Evaluación

```text
1. Aplicar Gates.
2. Revisar evidencia.
3. Validar con persona responsable.
4. Identificar riesgos pendientes.
5. Decidir: continuar, iterar, pausar, publicar o archivar.
```

## 9.5 Fase 5 — Cierre

```text
1. Crear cierre de sesión.
2. Actualizar changelog.
3. Actualizar registro de decisiones.
4. Actualizar primer de continuidad.
5. Actualizar wiki si aplica.
6. Archivar si corresponde.
```

# 10. Gates

Los Gates son compuertas de validación.

| Gate | Pregunta |
|---|---|
| G1 Propósito | ¿Está claro por qué existe el proyecto? |
| G2 Ubicación | ¿Dónde vive el trabajo dentro del sistema? |
| G3 Perfil / Caleidoscopio | ¿Se activó el lente correcto? |
| G4 Metodología | ¿Hay ruta de trabajo? |
| G5 Evidencia | ¿Qué demuestra el avance? |
| G6 IA responsable | ¿Cómo se usó y revisó IA? |
| G7 Ética y derechos | ¿Hay riesgos humanos, culturales o legales? |
| G8 Accesibilidad | ¿Puede usarse por audiencias diversas? |
| G9 Calidad | ¿Cumple criterios técnicos, investigativos o creativos? |
| G10 Documentación | ¿Quedó documentado? |
| G11 Wiki / transferencia | ¿Debe traducirse a guía humana? |
| G12 Archivo | ¿Debe preservarse? |
| G13 Continuidad | ¿Alguien puede retomarlo? |
| G14 Estabilización | ¿Puede declararse versión estable? |

# 11. CHECKS mínimos

## 11.1 CHECKS de inicio

- [ ] Propósito claro.
- [ ] Usuario o comunidad identificada.
- [ ] Producto esperado definido.
- [ ] Nivel N1-N4 seleccionado.
- [ ] Perfil o Caleidoscopio activado.
- [ ] Riesgos iniciales identificados.
- [ ] Evidencia mínima definida.
- [ ] Uso de IA definido.
- [ ] Cartapacio o espacio de trabajo definido.

## 11.2 CHECKS de producción

- [ ] Se produjo evidencia.
- [ ] Se documentaron decisiones.
- [ ] Se revisó ética.
- [ ] Se revisó accesibilidad.
- [ ] Se revisaron derechos/licencias.
- [ ] Se validó uso de IA.
- [ ] Se aplicaron criterios de calidad.
- [ ] Se identificaron pendientes.

## 11.3 CHECKS de cierre

- [ ] Producto tiene nombre, versión y estado.
- [ ] Hay cierre o resumen.
- [ ] Changelog actualizado si aplica.
- [ ] Registro de decisiones actualizado si aplica.
- [ ] Wiki actualizada o marcada pendiente si aplica.
- [ ] Archivo preparado si aplica.
- [ ] Primer de continuidad creado si habrá seguimiento.
- [ ] Próxima acción definida.

# 12. Uso responsable de IA

## 12.1 La IA puede apoyar en:

```text
síntesis;
redacción;
comparación;
generación de ideas;
programación asistida;
revisión de estructura;
creación de plantillas;
evaluación preliminar;
organización de documentos;
traducción de documentación;
detección de riesgos.
```

## 12.2 La IA no debe sustituir:

```text
decisión humana;
validación ética;
responsabilidad institucional;
lectura crítica;
consentimiento;
derechos culturales;
evaluación de impacto;
verificación de datos;
aprobación final.
```

## 12.3 Toda salida con IA debe registrar:

```text
rol de IA;
aporte humano;
validación realizada;
límites;
riesgos;
si requiere disclosure.
```

# 20. Criterios de éxito del framework

Un proyecto implementado con V4.1.0 debe poder responder:

```text
¿Qué necesidad atendió?
¿Por qué nivel se trabajó?
¿Qué perfil o integración se activó?
¿Qué evidencia produjo?
¿Qué riesgos se revisaron?
¿Cómo se usó IA?
¿Qué decisiones se tomaron?
¿Qué documentos se generaron?
¿Dónde están?
¿Cómo se retoma?
¿Qué se aprendió?
```

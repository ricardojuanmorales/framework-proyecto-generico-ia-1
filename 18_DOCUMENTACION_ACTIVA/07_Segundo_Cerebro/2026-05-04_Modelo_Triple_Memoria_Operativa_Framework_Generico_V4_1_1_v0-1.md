## 1. Resumen ejecutivo

El **Modelo Triple Memoria Operativa** organiza la continuidad del Framework Genérico V4.1.1 mediante tres espacios complementarios:

```text
ChatGPT / sesión
→ taller de pensamiento, producción, evaluación y cierre

GitHub / repositorio
→ fuente oficial versionada, auditable y preservable

Segundo cerebro
→ tablero humano, mapa interpretativo, agenda, reflexión y navegación cotidiana
```

A esta triple memoria se le añade un sistema de **Issues ligeros** para convertir tareas, decisiones pendientes, pilotos, refinamientos y riesgos en unidades monitoreables sin convertir el proyecto en una burocracia pesada.

El modelo permite que el Framework Genérico se gobierne a sí mismo usando su propia lógica: propósito, evidencia, perfiles, Caleidoscopio, Gates, CHECKS, WIKI_SYNC, archivo y continuidad.

---

## 2. Problema que atiende

El proyecto ha alcanzado un nivel de madurez donde ya no basta con producir documentos. Ahora necesita un flujo estable para:

```text
mantener contexto actualizado;
evitar pérdida de decisiones;
actualizar documentos de gobernanza;
monitorear evolución;
relacionar sesiones con repositorio;
relacionar repositorio con segundo cerebro;
preparar pilotos y futuras versiones;
evitar duplicación documental;
evitar conversación vapor;
evitar archivo mausoleo.
```

Sin este modelo, el framework corre tres riesgos:

1. **Fragmentación:** documentos, ideas y decisiones viven en lugares desconectados.
2. **Desfase:** el segundo cerebro dice una cosa, GitHub otra y la sesión recuerda otra.
3. **Sobrecarga:** cada actualización exige demasiado esfuerzo y el sistema deja de usarse.

---

## 3. Principio rector

```text
GitHub conserva la verdad oficial.
El segundo cerebro conserva la interpretación viva.
ChatGPT orquesta la producción, evaluación y continuidad.
Los Issues ligeros convierten intención en seguimiento.
```

La regla central:

```text
Nada importante debe vivir solo en una conversación.
Nada oficial debe vivir solo en el segundo cerebro.
Nada interpretativo debe depender solo del repositorio.
```

## 5. Función de cada espacio

## 5.1 ChatGPT / Sesión

Funciona como:

```text
laboratorio de pensamiento;
orquestador humano-IA;
redactor inicial;
evaluador;
sintetizador;
generador de documentos;
actualizador de continuidad.
```

Debe producir siempre una salida transferible:

```text
documento MD;
decisión;
changelog;
primer;
issue sugerido;
actualización wiki;
reporte de cierre;
archivo o release.
```

No debe ser la única memoria.

---

## 5.2 GitHub / Repositorio

Funciona como:

```text
fuente canónica;
archivo versionado;
estructura oficial;
historial auditable;
base de releases;
espacio de issues;
lugar de gobernanza.
```

Debe contener:

```text
README principal;
manifest;
índice maestro;
documentos oficiales;
changelog;
registro de decisiones;
primers;
wiki exportada;
releases;
archivo histórico.
```

GitHub responde:

```text
¿Qué existe oficialmente?
¿Cuál es la versión vigente?
¿Qué cambió?
¿Qué decisión lo autorizó?
Dónde vive cada documento?
```

---

## 5.3 Segundo cerebro

Funciona como:

```text
dashboard interpretativo;
memoria humana;
mapa de relaciones;
agenda de evolución;
sistema de reflexión;
registro de preguntas abiertas;
puente entre documentos y sentido.
```

Debe contener:

```text
estado actual;
versión activa;
próximos pasos;
issues importantes;
riesgos;
mapa de perfiles;
mapa Caleidoscopio;
notas de lectura;
ideas en incubación;
decisiones vivas con enlaces a GitHub.
```

El segundo cerebro no debe duplicar documentos completos salvo que sea necesario. Debe resumir, enlazar y orientar.

## 6. Issues ligeros

Los **Issues ligeros** son unidades mínimas de seguimiento. No son expedientes pesados. Son tarjetas de intención con suficiente contexto para actuar.

Cada issue debe responder:

```text
¿Qué hay que hacer?
¿Por qué importa?
¿Qué nivel N1-N4 aplica?
¿Qué perfil o modo Caleidoscopio se activa?
¿Qué documentos afecta?
¿Qué evidencia cerrará el issue?
Qué riesgo atiende?
```

Ejemplos de Issues:

```text
D-A45 — Crear wiki secundaria V4.1.1
D-A46 — Ejecutar primer piloto práctico
D-A47 — Crear matriz de riesgos ampliada
D-A48 — Refinar Investigador Transdisciplinario v0.2
D-A49 — Refinar Artista Transdisciplinario v0.2
D-A50 — Probar Caleidoscopio C1-C7 en proyecto real
```

## 7. Flujo operativo recomendado

## 7.1 Inicio de sesión

```text
1. Revisar Dashboard del segundo cerebro.
2. Abrir README del repositorio.
3. Abrir Primer de Continuidad más reciente.
4. Revisar Issues activas.
5. Definir objetivo de la sesión.
6. Declarar nivel N1-N4.
7. Activar perfil o Caleidoscopio.
```

## 7.2 Durante la sesión

```text
1. Producir o revisar documentos.
2. Aplicar Gates.
3. Aplicar CHECKS.
4. Registrar decisiones candidatas.
5. Identificar documentos afectados.
6. Determinar si WIKI_SYNC aplica.
7. Identificar si se requiere issue nuevo.
```

## 7.3 Cierre de sesión

```text
1. Actualizar Changelog.
2. Actualizar Registro de Decisiones.
3. Actualizar Primer.
4. Actualizar Manifest si cambió estructura.
5. Actualizar Índice si se creó documento.
6. Actualizar Dashboard del segundo cerebro.
7. Cerrar o crear Issues.
8. Preparar commit o release.
```

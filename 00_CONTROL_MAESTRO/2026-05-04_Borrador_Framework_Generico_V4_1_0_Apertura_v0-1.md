## 0. Declaración de apertura

Se abre formalmente el **borrador del Framework Genérico V4.1.0** como evolución controlada del Framework Genérico V4.0.0.

Esta apertura no representa una reescritura total ni una sustitución inmediata de la V4.0.0. Representa la activación de una rama evolutiva destinada a integrar formalmente la capa de **Perfiles Transversales y Caleidoscopio**, junto con sus matrices, ejemplos operativos, plantillas base y documentos de gobernanza asociados.

La V4.0.0 queda preservada como arquitectura madre estabilizada y base histórica. La V4.1.0 se abre como borrador de actualización, integración y operacionalización.

**Principio de transición:**

> V4.1.0 no nace para borrar V4.0.0, sino para hacerla más operable, transversal, documentable, transferible y preparada para proyectos complejos humano-IA.

---

## 1. Resumen ejecutivo

El Framework Genérico V4.0.0 alcanzó un estado de madurez suficiente para pasar de congelamiento protector a reactivación controlada. Durante el ciclo previo se consolidó una capa transversal compuesta por:

1. **Programador Humanista v0.2**, cerrado funcionalmente.
2. **Investigador Transdisciplinario v0.1**, formalizado preliminarmente.
3. **Artista Transdisciplinario v0.1**, formalizado preliminarmente.
4. **Caleidoscopio v0.1**, formalizado como lógica integradora emergente.

Además, se generaron documentos derivados que convierten esa capa en una arquitectura operable:

1. Paquete consolidado de perfiles y Caleidoscopio.
2. Matriz comparativa transversal.
3. Matriz Caleidoscopio C1-C7 con ejemplos operativos.
4. Plantillas base `agents`, `skills`, `workflows`, `checks` y `wiki_sync`.
5. Reporte de cierre de situación actual V4.0.0.
6. Changelog de actualización.
7. Registro de decisiones actualizado.
8. Primer de continuidad para V4.1.0-borrador.

La V4.1.0 debe integrar estos avances sin inflar la arquitectura, sin crear macrocarpetas innecesarias y sin reabrir perfiles cerrados sin decisión explícita.

---

## 2. Propósito de la V4.1.0

El propósito de la V4.1.0 es convertir el Framework Genérico V4.0.0 en una arquitectura más operativa para proyectos transdisciplinarios, humano-IA y documentales que requieren:

- perfiles transversales activables;
- integración entre investigación, programación y creación;
- Caleidoscopio como lógica emergente;
- documentación activa desde el inicio;
- agentes, skills, workflows y checks reutilizables;
- sincronización con wiki humana;
- evaluación multidimensional;
- gobernanza ética, técnica, cultural y socioambiental;
- cierre, archivo y transferencia.

La actualización busca mejorar el paso de arquitectura conceptual a operación situada.

## 4. Decisión de apertura

```text
D-A28 — Descongelar de forma controlada el Framework Genérico V4.0.0 y abrir el borrador Framework Genérico V4.1.0 para integrar formalmente la capa de Perfiles Transversales, Caleidoscopio, matrices, ejemplos operativos, plantillas base y gobernanza documental asociada.
```

**Estado:** aceptada por validación humana.  
**Tipo de cambio:** cambio metodológico y de gobernanza con extensión modular.  
**Impacto:** alto, pero controlado.  
**Riesgo:** medio si se confunde apertura con reescritura. Bajo si se respeta la ruta de transición.  
**Documentos afectados:** README, changelog, registro de decisiones, primer de continuidad, estado actual, documentación activa, wiki, archivo histórico.  
**Recomendación:** ejecutar en fases, con cierre por productos.

## 6. Núcleo conceptual que V4.1.0 debe preservar

La V4.1.0 conserva los principios de V4.0.0:

1. Juicio humano como centro de decisión.
2. IA como mediación, no autoridad.
3. Documentación activa desde el inicio.
4. Ética, accesibilidad, evaluación y trazabilidad.
5. Cierre, archivo y continuidad como parte del trabajo.
6. Modularidad sin proliferación.
7. Transferencia y reutilización responsable.
8. Desarrollo humano integral.
9. Pluralismo epistemológico.
10. Gobernanza técnica, ética y documental.

La V4.1.0 añade una capa más operativa para activar esos principios en proyectos concretos.

La V4.1.0 adopta el Caleidoscopio como lógica de integración emergente.

### 8.1 Modos C1-C7

| Modo | Combinación | Uso principal |
|---|---|---|
| C1 | Programador + Investigador | Herramientas, dashboards, bibliometría, repositorios. |
| C2 | Investigador + Artista | Investigación-creación, memoria, publicación multimodal. |
| C3 | Artista + Programador | Arte digital, galerías, avatares, interfaces. |
| C4 | Programador + Investigador + Artista | AVAT, REA multimodal, ecosistemas agénticos. |

| C7 | Institucional | Laboratorios, políticas, manuales, wiki y capacitación. |

### 8.2 Regla Caleidoscopio

> Ninguna combinación se considera válida si no produce evidencia, integración real, documentación, evaluación, accesibilidad, derechos/licencias y continuidad.

## 10. Reglas de protección V4.1.0

```text
1. No reescribir perfiles cerrados sin decisión explícita.
2. No crear macrocarpetas nuevas sin análisis de impacto.
3. Primero mapa, luego documento, luego carpeta, luego automatización.
4. Todo cambio debe clasificarse.
5. Todo cambio debe incluir justificación, impacto, riesgo, documentos afectados, versión y recomendación.
6. Wiki, cierre y archivo son parte del trabajo, no tareas posteriores.
7. Caleidoscopio no se reduce a suma de perfiles.
8. IA no sustituye validación humana.
9. Toda salida debe tener evidencia mínima y Gate aplicable.
10. Toda versión debe dejar continuidad.
```

## 15. Riesgos de la transición

| Riesgo | Descripción | Salvaguarda |
|---|---|---|
| Reescritura total | Intentar reconstruir V4.0.0 desde cero. | V4.1.0 como rama evolutiva, no sustitución inmediata. |
| Inflación documental | Crear documentos sin uso operativo. | Cada documento debe tener propósito, Gate y continuidad. |
| Macrocarpetas innecesarias | Expandir estructura sin necesidad. | No crear nuevas macrocarpetas. Integrar en cartapacios existentes. |
| Caleidoscopio superficial | Usarlo como suma de perfiles. | Exigir emergencia, evidencia y validación integrada. |
| Automatización prematura | Crear agentes ejecutables sin gobernanza. | Plantillas primero; automatización después. |
| Pérdida de trazabilidad | No registrar cambios de versión. | Changelog, decisiones, manifest y primer. |
| Wiki desactualizada | La documentación humana queda atrás. | WIKI_SYNC obligatorio. |
| Evaluación tardía | Evaluar al final. | Gates desde la apertura. |
| Ambigüedad v0.1/v0.2/v4.1.0 | Confundir versiones de perfiles con versión del framework. | Tabla de versiones y manifest. |

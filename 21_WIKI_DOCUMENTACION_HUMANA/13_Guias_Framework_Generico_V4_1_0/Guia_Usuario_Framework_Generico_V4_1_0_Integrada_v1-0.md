# 1. Resumen ejecutivo

El **Framework Genérico V4.1.0** es un marco de trabajo para iniciar, organizar, desarrollar, evaluar, documentar, archivar y transferir proyectos apoyados por inteligencia artificial, investigación, creación, programación y colaboración humano-máquina.

Está diseñado para personas y equipos que necesitan convertir una idea, necesidad, problema o posibilidad en un proceso operativo con:

```text
propósito claro;
nivel de activación;
perfil de trabajo;
alcance;
evidencia;
evaluación;
documentación;
gobernanza;
cierre;
continuidad.
```

El framework puede usarse en proyectos:

```text
educativos;
investigativos;
artísticos;
tecnológicos;
comunitarios;
institucionales;
curriculares;
culturales;
administrativos;
multimodales;
de innovación con IA.
```

Esta guía ayuda al usuario a responder cinco preguntas fundamentales:

```text
1. ¿Qué necesito hacer?
2. ¿Qué nivel de complejidad tiene mi proyecto?
3. ¿Qué perfil transversal debo activar?
4. ¿Necesito usar el lente Caleidoscopio?
5. ¿Qué documentos, evidencias y pasos debo completar para avanzar?
```

---

# 2. Propósito de la guía

Esta guía sirve para que cualquier usuario pueda activar el Framework Genérico V4.1.0 sin tener que conocer toda la historia del framework.

Su propósito es:

```text
orientar el inicio de proyectos nuevos;
explicar niveles N1-N4;
explicar perfiles transversales;
explicar el lente Caleidoscopio;
proponer rutas de trabajo;
identificar documentos mínimos;
evitar uso excesivo del framework;
evitar subdocumentar proyectos complejos;
facilitar cierre y continuidad.
```

---

# 3. ¿Cuándo usar el Framework Genérico V4.1.0?

Usa este framework cuando necesites:

```text
organizar un proyecto;
usar IA de forma responsable;
crear un recurso educativo;
diseñar una investigación;
desarrollar un prototipo;
crear una obra o experiencia multimodal;
documentar un proceso;
trabajar con un equipo;
crear una wiki;
preparar un archivo;
dar continuidad entre sesiones;
evaluar riesgos;
preparar transferencia o adopción.
```

No necesitas usar todo el framework siempre.  
La clave es seleccionar el nivel adecuado.

# 4. Diagnóstico inicial del usuario

Antes de activar un nivel o perfil, responde:

```text
1. ¿Qué problema, necesidad o deseo quieres atender?
2. ¿Quiénes se benefician o se ven afectados?
3. ¿Qué producto, resultado o cambio esperas lograr?
4. ¿El proyecto requiere investigación?
5. ¿El proyecto requiere creación artística, narrativa o multimodal?
6. ¿El proyecto requiere programación, repositorio, automatización o herramienta técnica?
7. ¿El proyecto requiere enseñanza, curso, módulo o actividad educativa?
8. ¿El proyecto involucra comunidad, cultura, memoria, territorio o patrimonio?
9. ¿Usarás inteligencia artificial? ¿Para qué?
10. ¿Hay datos sensibles, menores, imágenes, voces, historias personales o derechos culturales?
11. ¿Cuánto durará el proyecto?
12. ¿Trabajará una persona, un equipo pequeño, una comunidad o una institución?
13. ¿Qué evidencia demostrará avance?
14. ¿Quién debe poder continuar el trabajo después?
```

# 5. Niveles de activación N1-N4

El framework usa cuatro niveles de activación para ajustar la profundidad del trabajo.

```text
N1 = mínimo controlado
N2 = operativo
N3 = avanzado
N4 = institucional
```

La idea es sencilla:

```text
usar lo suficiente para sostener calidad,
pero no tanto que el proceso se vuelva una jaula documental.
```

---

# 6. Nivel N1 — Mínimo controlado

## 6.1 Descripción

El nivel **N1** es para tareas pequeñas, exploratorias, de bajo riesgo o de una sola sesión.

Se usa cuando el proyecto no requiere equipo formal, archivo complejo, publicación pública ni gobernanza amplia.

## 6.2 Cuándo usar N1

Usa N1 cuando:

```text
la tarea es breve;
el riesgo es bajo;
trabaja una sola persona;
no hay datos sensibles;
no hay publicación pública importante;
no se requiere continuidad larga;
no hay múltiples perfiles activos.
```

## 6.3 Objetivos operacionales

Al usar N1, debes lograr:

```text
definir propósito;
producir un resultado simple;
registrar una nota breve de cierre;
identificar si el trabajo debe subir a N2.
```

## 6.4 Documentos mínimos

```text
brief breve;
producto generado;
nota de cierre;
próximo paso si aplica.
```

## 6.5 Ejemplos

```text
crear una actividad corta;
organizar ideas para un proyecto;
generar una lista de criterios;
preparar una explicación sencilla;
crear una plantilla básica.
```

## 6.6 Criterio de cierre N1

Un trabajo N1 se puede cerrar si responde:

```text
¿Qué se hizo?
¿Por qué se hizo?
¿Dónde quedó?
¿Hace falta continuar?
```

---

# 7. Nivel N2 — Operativo

## 7.1 Descripción

El nivel **N2** es para proyectos con varias fases, seguimiento, uso moderado de IA o colaboración de una persona con apoyo externo o un equipo pequeño.

N2 introduce documentación mínima para que el trabajo pueda retomarse.

## 7.2 Cuándo usar N2

Usa N2 cuando:

```text
el proyecto tendrá más de una sesión;
hay equipo pequeño;
hay documentos reutilizables;
hay uso de IA que debe explicarse;
hay necesidad de seguimiento;
el producto será compartido con otras personas;
se requiere una estructura básica.
```

## 7.3 Objetivos operacionales

Al usar N2, debes lograr:

```text
crear brief de proyecto;
definir alcance;
seleccionar perfil activo;
establecer evidencia mínima;
registrar decisiones básicas;
crear cierre de sesión;
dejar continuidad.
```

## 7.4 Documentos mínimos

```text
brief de proyecto;
README o nota de orientación;
checklist básico;
registro breve de decisiones;
cierre de sesión;
primer de continuidad si habrá seguimiento.
```

## 7.5 Ejemplos

```text
crear un módulo educativo;
preparar un taller;
organizar una pequeña investigación;
crear un prototipo simple;
desarrollar una guía para colegas;
crear una wiki mínima.
```

## 7.6 Criterio de cierre N2

Un trabajo N2 se puede cerrar si responde:

```text
¿Cuál fue el alcance?
¿Qué producto se generó?
¿Qué decisiones se tomaron?
¿Qué evidencia existe?
¿Qué queda pendiente?
¿Cómo se retoma?
```

---

# 8. Nivel N3 — Avanzado

## 8.1 Descripción

El nivel **N3** es para proyectos complejos que integran perfiles, IA sustantiva, riesgos éticos, publicación, comunidad, wiki, archivo o transferencia.

N3 es el nivel recomendado para proyectos transdisciplinarios completos.

## 8.2 Cuándo usar N3

Usa N3 cuando:

```text
hay investigación + creación + tecnología;
hay uso sustantivo de IA;
hay varios perfiles activos;
se requiere Caleidoscopio;
hay riesgos éticos, culturales, sociales o de datos;
hay publicación o transferencia;
hay wiki;
hay archivo histórico;
hay necesidad de trazabilidad.
```

## 8.3 Objetivos operacionales

Al usar N3, debes lograr:

```text
formalizar propósito, alcance y nivel;
activar perfil o Caleidoscopio;
crear ruta de trabajo;
aplicar Gates;
aplicar CHECKS;
registrar decisiones;
actualizar changelog;
documentar uso de IA;
revisar ética, derechos y accesibilidad;
crear wiki o WIKI_SYNC si aplica;
preparar archivo y continuidad.
```

## 8.4 Documentos mínimos

```text
brief completo;
README;
matriz de nivel y alcance;
Gates;
CHECKS;
registro de decisiones;
changelog;
primer de continuidad;
documentación de IA;
matriz de riesgos si aplica;
wiki o guía humana;
cierre de sesión;
archivo o release si aplica.
```

## 8.5 Ejemplos

```text
crear un curso completo con IA;
desarrollar una investigación con publicación multimodal;
crear un avatar transdisciplinario;
desarrollar un repositorio vivo;
crear una experiencia artística-documental;
implementar un piloto institucional;
crear un sistema de agentes o skills;
documentar un laboratorio.
```

## 8.6 Criterio de cierre N3

Un trabajo N3 se puede cerrar si responde:

```text
¿Qué necesidad compleja atendió?
¿Qué perfiles se activaron?
¿Qué evidencia se produjo?
¿Qué riesgos se revisaron?
¿Cómo se usó IA?
¿Qué documentos se actualizaron?
¿Qué queda en wiki?
¿Qué se archivó?
¿Cómo continúa?
```

# 9. Nivel N4 — Institucional

## 9.1 Descripción

El nivel **N4** es para adopción institucional formal, automatización, auditoría, capacitación, mantenimiento y escalamiento.

N4 no debe activarse por entusiasmo. Requiere responsables, gobernanza, recursos y continuidad.

## 9.2 Cuándo usar N4

Usa N4 cuando:

```text
hay adopción institucional;
hay muchas personas afectadas;
hay políticas o procedimientos;
hay automatización formal;
hay capacitación;
hay auditoría;
hay mantenimiento asignado;
hay riesgos legales, administrativos o de datos;
hay integración con sistemas institucionales.
```

## 9.3 Objetivos operacionales

Al usar N4, debes lograr:

```text
definir gobernanza formal;
asignar roles y responsabilidades;
crear RACI;
crear plan de capacitación;
crear política de uso de IA si aplica;
crear auditoría;
definir mantenimiento;
establecer soporte;
medir impacto;
preparar release institucional;
documentar riesgos y cumplimiento.
```

## 9.4 Documentos mínimos

```text
gobernanza formal;
RACI;
plan de implementación;
plan de capacitación;
matriz de riesgos ampliada;
política de IA si aplica;
plan de auditoría;
plan de mantenimiento;
release institucional;
archivo histórico;
dashboard de seguimiento.
```

## 9.5 Ejemplos

```text
adopción de IA en una facultad;
implementación de laboratorio institucional;
sistema de agentes para una oficina;
programa curricular completo;
repositorio institucional de recursos;
protocolo de evaluación docente con IA;
sistema de archivo histórico digital.
```

## 9.6 Criterio de cierre N4

Un trabajo N4 solo puede cerrarse si existe:

```text
aprobación humana responsable;
documentación institucional;
plan de mantenimiento;
auditoría o evaluación;
capacitación;
archivo;
continuidad formal.
```

# 11. Perfiles Transversales

Los perfiles transversales son lentes de trabajo. Ayudan a decidir qué tipo de pensamiento, evidencia, riesgos y productos necesita un proyecto.

Los perfiles principales son:

```text
Programador Humanista
Investigador Transdisciplinario
Artista Transdisciplinario
Caleidoscopio
```

Los primeros tres son perfiles operativos.  
El Caleidoscopio es un lente integrador.

---

# 12. Programador Humanista

## 12.1 Descripción

El **Programador Humanista** se activa cuando el proyecto requiere construcción técnica, código, repositorios, automatización, prototipos, sistemas, agentes, pruebas o despliegue.

No se limita a programar. Integra:

```text
problema;
contexto;
arquitectura;
diseño;
código;
pruebas;
seguridad;
documentación;
mantenimiento;
transferencia.
```

## 12.2 Cuándo activarlo

```text
hay código;
hay repositorio;
hay automatización;
hay datos;
hay prototipo;
hay herramienta digital;
hay agentes o skills;
hay integración técnica;
hay despliegue;
hay mantenimiento.
```

## 12.3 Objetivos operacionales

```text
traducir necesidades en requisitos;
definir arquitectura;
crear solución técnica;
documentar dependencias;
probar funcionamiento;
revisar seguridad;
preparar mantenimiento;
hacer transferible el sistema.
```

## 12.4 Evidencia esperada

```text
README técnico;
código o script;
diagrama;
tests;
workflow;
release;
registro de decisiones técnicas;
guía de instalación o uso.
```

## 12.5 Riesgos a revisar

```text
automatización prematura;
código sin pruebas;
dependencias no documentadas;
datos sensibles expuestos;
seguridad débil;
IA usada sin revisión;
falta de mantenimiento.
```

# 13. Investigador Transdisciplinario

## 13.1 Descripción

El **Investigador Transdisciplinario** se activa cuando el proyecto requiere preguntas, fuentes, evidencia, marcos conceptuales, metodología, análisis, síntesis o publicación.

No se limita a buscar información. Integra:

```text
problematización;
contexto;
saberes;
fuentes;
método;
evidencia;
análisis;
validación;
publicación;
archivo;
transferencia.
```

## 13.2 Cuándo activarlo

```text
hay pregunta de investigación;
hay revisión de literatura;
hay fuentes;
hay bibliografía;
hay análisis conceptual;
hay evidencia;
hay metodología;
hay publicación;
hay REA;
hay transferencia de conocimiento.
```

## 13.3 Objetivos operacionales

```text
formular problema investigable;
organizar fuentes;
definir criterios;
analizar evidencia;
sintetizar hallazgos;
validar interpretación;
documentar límites;
preparar comunicación o publicación.
```

## 13.4 Evidencia esperada

```text
pregunta guía;
matriz de fuentes;
bibliografía anotada;
evidence ledger;
marco conceptual;
nota metodológica;
síntesis;
reporte o publicación.
```

## 13.5 Riesgos a revisar

```text
fuentes débiles;
IA como autoridad;
sesgos;
falta de citación;
confundir dato con interpretación;
extracción de saberes sin reciprocidad;
síntesis sin validación.
```

hay símbolos;

## 14.3 Objetivos operacionales

prototipar;

# 15. Lente Caleidoscopio

## 15.1 Descripción

El **Caleidoscopio** es la lógica integradora del framework. Se activa cuando un solo perfil no basta.

No es un perfil tradicional. Es un lente para integrar investigación, creación, programación, pedagogía, comunidad o institución.

## 15.2 Pregunta central

```text
¿Qué emerge cuando programación, investigación y creación se transforman mutuamente mediante evidencia, ética, accesibilidad, documentación y transferencia?
```

## 15.3 Cuándo activarlo

Activar Caleidoscopio cuando:

```text
el problema es complejo;
hay más de un perfil necesario;
hay producto multimodal;
hay comunidad o institución;
hay IA sustantiva;
hay riesgos culturales, éticos o socioambientales;
hay necesidad de transferencia;
hay archivo vivo;
hay combinación entre evidencia, técnica y experiencia.
```

## 15.4 Modos C1-C7

| Modo | Combinación | Uso típico |
|---|---|---|
| C1 Técnico-investigativo | Programador + Investigador | dashboards, apps de investigación, pipelines |
| C2 Investigativo-creativo | Investigador + Artista | memoria, investigación-creación, publicación multimodal |
| C3 Creativo-técnico | Artista + Programador | avatares, interfaces, galerías digitales |
| C4 Integral | Tres perfiles | AVAT, REA multimodal, ecosistemas agénticos |
| C5 Pedagógico | perfiles según necesidad | cursos, módulos, OVAT |
| C6 Comunitario | comunidad + perfiles | archivo vivo, memoria territorial |
| C7 Institucional | perfiles + gobernanza | laboratorios, programas, políticas |

## 15.5 Objetivos operacionales

```text
definir problema complejo;
mapear contexto;
activar perfiles necesarios;
seleccionar modo C1-C7;
diseñar workflow integrado;
producir evidencia;
evaluar emergencia;
documentar integración;
preparar transferencia;
archivar continuidad.
```

## 15.6 Evidencia esperada

```text
brief complejo;
matriz de perfiles;
modo C1-C7 justificado;
workflow integrado;
producto emergente;
matriz de riesgos;
CHECKS de accesibilidad;
documentación y archivo.
```

# 16. Cómo decidir entre perfil y Caleidoscopio

| Si el proyecto requiere principalmente... | Activar |
|---|---|
| código, repositorio, automatización o sistema | Programador Humanista |
| fuentes, evidencia, investigación o síntesis | Investigador Transdisciplinario |
| obra, narrativa, memoria, cultura o multimodalidad | Artista Transdisciplinario |
| integración entre varios de los anteriores | Caleidoscopio |
| adopción formal, política, auditoría o capacitación | N4 + C7 Institucional |

Regla:

```text
si un perfil basta, usar perfil;
si dos o más perfiles se transforman mutuamente, usar Caleidoscopio.
```

# 17. Ruta de uso recomendada

## Paso 1 — Diagnosticar

Completar preguntas iniciales.

## Paso 2 — Seleccionar nivel

Elegir N1, N2, N3 o N4.

## Paso 3 — Activar perfil o Caleidoscopio

Elegir lente de trabajo.

## Paso 4 — Crear brief

Documentar propósito, contexto, producto y evidencia.

## Paso 5 — Trabajar por fases

Producir, revisar, documentar y ajustar.

## Paso 6 — Aplicar Gates y CHECKS

Verificar si se puede avanzar.

## Paso 7 — Cerrar

Crear cierre, registrar decisiones y dejar continuidad.

## Paso 8 — Archivar o transferir

Preparar wiki, archivo, release o siguiente ciclo.

# 23. Cierre

El Framework Genérico V4.1.0 ayuda a que los proyectos no dependan solo de inspiración, urgencia o memoria dispersa.

Ofrece:

```text
mapa;
niveles;
perfiles;
lente integrador;
Gates;
CHECKS;
documentación;
archivo;
continuidad.
```

Su mejor uso no es llenar carpetas.  
Su mejor uso es crear proyectos que puedan entenderse, evaluarse, cuidarse, compartirse y continuar.

**Estado de esta guía:** lista para uso por usuarios y colaboradores.
